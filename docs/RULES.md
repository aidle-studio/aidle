# RULES.md

## 1. 開発原則
- 仕様駆動開発（SDD）: 仕様は `docs/SPEC.md` を正本とし、`REQ-*` / `AC-*` で管理する。
- テスト駆動開発（TDD）: `Red -> Green -> Refactor` を必須サイクルとする。
- 合意ゲート: 仕様変更は合意後にのみ実装へ進める。

## 2. AI駆動開発の標準手順
1. マイルストーンを定義し、達成タスクへ細分化する。
2. マイルストーン達成のためのプランを作成し、合意を得る。
3. 合意したプランに従って実行する。

## 3. コンテキスト更新ルール
- セッションの開始時または状態に変化があった場合、AIエージェントは自律的に `docs/AGENT_CONTEXT.md` の「現在フェーズ」と「直近ゴール」を最新状態に更新すること。
- 次回のアクションが明確になった時点で、「次アクション」のリストを更新し、コンテキストの連続性を維持すること。

## 4. 反復実行ルール
- マイルストーン達成までの反復実行は、事前に合意がある場合のみ行う。
- 反復ごとに進捗・差分・次アクションを更新する。
- 想定外の論点が出た場合は、続行前に再合意する。

## 5. プラン保存ルール
- 作成したプランは `docs/plans/` 配下にファイルとして保存する。
- ファイル名は `YYYY-MM-DD_<milestone-id>.md` 形式を推奨する。
- 途中再開時は、同一プランファイルを更新して履歴を残す。
- 新規プランは `docs/PLAN_TEMPLATE.md` を雛形として作成する。

## 6. TDDサイクル（必須）
1. Red: 失敗するテストを先に追加する。
2. Green: テストを通す最小実装を追加する。
3. Refactor: 挙動を変えずに整理する。

## 7. テスト状態ルール
- テスト状態は `Red` / `Provisional Green` / `Green` の3状態で管理する。
- `Provisional Green` は疎通目的の暫定実装を許可するが、完了扱いにはしない。
- `Provisional Green` の項目は、理由・解消期限・担当を `TODO` または plan に必ず記録する。
- `Provisional Green` は次フェーズ開始前に解消必須とし、持ち越しは例外申請がある場合のみ許可する。

## 8. テストルール
- テストIDは `TC-*` で管理し、必ず `AC-*` と対応付ける。
- テストランナーは `cargo nextest` を標準とする。
- 正常系だけでなく、境界値または異常系を最低1件以上追加する。
- バグ修正は再現テスト（Red）を先に追加してから実装する。
- タイムアウトは `nextest.toml` で管理する:
  - default: 15s
  - slow: 45s
  - very_slow: 90s（例外、理由必須）

## 9. 品質ゲート
- 行カバレッジ: 全体 80%以上
- 差分カバレッジ: 目標 90%以上（未達時は理由をPRに記載）
- 重要フロー（init正常系、dry-run、エラー契約、ロールバック）は統合テスト必須
- 実行手順は `docs/COVERAGE_RUNBOOK.md` を正本とする
- 例外申請は `docs/COVERAGE_EXCEPTION_TEMPLATE.md` を使用する
- CI品質ゲートは `.github/workflows/coverage-gate.yml` を正本とする
- `cargo clippy -- -D warnings` と `cargo fmt -- --check` を完了条件に含める。

## 10. トレーサビリティ
- 各 `AC-*` は対応する `TC-*` と実装差分に紐付いていること。
- 仕様変更時は、対応する `TC-*` と `TODO` / plan を同一PRで更新すること。

## 11. レビュー基準
- 仕様・テスト・実装のトレーサビリティがあること
- 破壊的変更は明示され、影響範囲が説明されていること
- 例外運用には期限付きの解消計画があること
- 入力依存で挙動が変わる実装になっていること（固定値依存の偽Greenを禁止）
- テストが実装詳細に過度依存していないこと

## 12. ドキュメント同期ルール
- 状態変更時は `docs/TODO.md` / `docs/TEST_PLAN.md` / `docs/plans/*.md` を同一PRで同期更新する。

## 13. コミット規約
- 基本形式は「タイトル1行」とする。
- タイトルは何を変更したかが分かる短い文にする（目安: 50文字前後、最大72文字程度）。
- バグ再現手順、設計トレードオフ、判断理由など特筆事項がある場合は、タイトル + 本文の形式にする。
- 本文では「何を変えたか / なぜ必要か / 実装上の注意点」を明記する。
- バグ修正では可能な限り再現手順を本文に含める。
- コミットメッセージの言語は日本語・英語のどちらでも可とする（英語強制はしない）。
- 1コミット内では、読みやすさのため言語の混在を最小限にする。

## 14. Definition of Done
- 対応する `TC-*` が `Green` である（`Provisional Green` は完了不可）
- 影響範囲のテストがすべて成功している
- 仕様との差分がない（必要なら仕様更新済み）
- `cargo clippy -- -D warnings` / `cargo fmt -- --check` が成功している
- 変更理由がコミットまたはPR説明で追跡可能である
