## ADR-0001: CLI実装言語としてRustを採用

- 日付: 2026-03-08
- ステータス: Accepted
- 決定者: プロジェクトチーム

### Context
- `aidle` はプロジェクト雛形を生成するCLIとして、OS間で安定して動作することが重要。
- 利用者の環境差異（ランタイム有無、バージョン不一致）による導入障壁を下げたい。
- 生成処理は多数のファイル作成を伴うため、起動速度と実行性能も重視する。

### Decision
- CLIの実装言語としてRustを採用する。

### Consequences
- メリット:
  - 単一バイナリ配布によってランタイム依存を減らせる。
  - 高速かつ安定したCLI実行が可能。
  - 型安全な実装で品質を担保しやすい。
- デメリット:
  - メンバーによっては学習コストが発生する。
  - ビルド時間が相対的に長くなる場合がある。

### Follow-ups
- 初期実装で使用するライブラリを確定する（`clap`, `serde`, テンプレートエンジン）。
- TDD方針に合わせ、CLI振る舞いの統合テスト方針を `TEST_PLAN.md` に反映する。

## ADR-0002: 初期リリースのテンプレート対応範囲

- 日付: 2026-03-08
- ステータス: Accepted
- 決定者: プロジェクトチーム

### Context
- 早期に品質の高い `init` 体験を確立するためには、初期対応テンプレートを絞る必要がある。
- 初期段階でテンプレートを広げすぎると、テスト負荷と保守コストが急増する。

### Decision
- 初期リリース（v0.x）では公式テンプレートを `default` の1種類に限定する。
- `default` は AIDD 用ドキュメント構成（`AGENTS.md`, `docs/*`）を必須生成対象とする。
- 言語・フレームワーク別テンプレートは v0.x 後半で段階追加する。

### Consequences
- メリット:
  - 仕様とテストを集中でき、安定性を高めやすい。
  - ロールバックやエラー処理を先に固められる。
- デメリット:
  - 初期は言語別最適化が限定的になる。

### Follow-ups
- テンプレート追加時は ADR を追加し、受け入れ基準と `TEST_PLAN.md` を更新する。

## ADR-0003: 設定ファイルフォーマットにTOMLを採用

- 日付: 2026-03-08
- ステータス: Accepted
- 決定者: プロジェクトチーム

### Context
- `aidle` は Rust 実装であり、設定の読み書きで開発体験と保守性を重視する。
- 設定は人間が手で編集するケースが多いため、可読性とコメント運用も重要。

### Decision
- `aidle` の主要設定ファイルフォーマットとして TOML を採用する。
- 初期設定ファイル名は `aidle.toml` とする。

### Consequences
- メリット:
  - Rust エコシステムとの親和性が高い。
  - 設定構造を明確に保ちやすい。
- デメリット:
  - YAML 利用者には慣れが必要。

### Follow-ups
- `aidle.toml` のスキーマ初版を `SPEC.md` または専用ドキュメントへ追加する。
## ADR-0004: 対話プロンプト用ライブラリとして `dialoguer` を採用

- 日付: 2026-03-09
- ステータス: Accepted
- 決定者: プロジェクトチーム

### Context (背景)
- MS9 において、人間が `aidle init` を実行した際（TTY接続時）に、不足している情報を対話形式で入力できるウィザード機能が必要になった。
- 将来的（MS10等）には、「言語の選択」や「AIの役割選択」といった複数選択（Select/MultiSelect）のプロンプトも必要になることが想定される。
- Rustエコシステムにおける対話プロンプトの主要な選択肢として、`dialoguer` と `inquire` が存在する。

### Decision (決定事項)
- 対話プロンプト用のライブラリとして `dialoguer` を採用する。

### Consequences (結果・影響)
- メリット:
  - 広く使われており、安定している（mitsuhiko氏のライブラリであり、`console` などのエコシステムとの連携が強力）。
  - `Select` や `MultiSelect`、`Input` といった必要な機能がシンプルかつ十分に揃っている。
  - テスト時に擬似的に入力を流し込むような仕組みを作りやすい。
- デメリット / リスク:
  - UIのカスタマイズ性（テーマ機能など）は `inquire` の方がリッチである場合がある。しかし、`aidle` はシンプルなCLIツールであるため、過度な装飾は不要と判断した。

### Follow-ups (次回アクション)
- `Cargo.toml` に `dialoguer` を追加し、TTY判定とプロンプトの実装を進める。
