# AI駆動開発・仕様駆動開発・テスト駆動開発 ベストプラクティス調査ノート

AI駆動開発（AIDD）において、AIのパフォーマンスを最大化しつつ、品質を担保するためのベストプラクティスを調査・整理しました。これらの知見は `aidle` が生成するテンプレートの設計思想に直結します。

## 1. AI駆動開発 (AI-Driven Development: AIDD)
AI（LLM）をペアプログラマーや自律的エージェントとして活用し、開発効率と品質を最大化するアプローチ。

### ベストプラクティス
- **文脈（Context）の明示的提供:**
  AIは文脈がないと一般的な推測でコードを生成してしまうため、プロジェクトの規約、ディレクトリ構造、現在のフェーズを明記したドキュメント（`RULES.md`など）を常に読ませる仕組みが重要。
- **ドキュメントベースでの振る舞い制御:**
  毎回長文のプロンプトを入力するのではなく、ワークスペース内のマークダウン（`.md`）でルールを管理し、AIにそれを参照させる（`AGENTS.md` のような正本）。
- **役割（Role）の付与:**
  AIに対して「シニアアーキテクト」「厳格なテストエンジニア」などの役割を与えることで、出力の質と方向性をコントロールする。
- **ステップ・バイ・ステップの進行:**
  一度に大量のコード変更を依頼せず、「仕様策定」→「テスト作成」→「実装」と段階を追ってタスクを分割する。

## 2. 仕様駆動開発 (Specification-Driven Development: SDD)
実装の前に「何を作るべきか（仕様）」を明確に定義し、それに従って開発を進める手法。AI開発においては「AIへの指示書」となる。

### ベストプラクティス
- **「仕様が先、コードは後」の徹底:**
  いきなりAIにコードを書かせる（雰囲気実装 / Vibe Coding）のではなく、まずは要件定義と仕様を確定させる。
- **人間がレビュー可能なフォーマットでの合意:**
  AIが出力した設計や仕様を人間が確認し、「合意（Agreement Gate）」を形成してから次の工程に進む。
- **仕様の細分化:**
  巨大な仕様書はAIのコンテキストウィンドウを圧迫し、コンテキストロストやハルシネーションの原因になるため、小さなタスク単位に分解する。
- **受け入れ基準 (Acceptance Criteria: AC) の明確化:**
  「何ができたら完了か」を箇条書き（`AC-001`, `AC-002` など）で明確に定義し、テストケースの入力とする。

## 3. テスト駆動開発 (Test-Driven Development: TDD)
テストを先に書き、そのテストを通すための最小限の実装を行い、最後にリファクタリングするサイクル（Red -> Green -> Refactor）を回す手法。

### AI環境におけるTDDのベストプラクティス
- **Red-Green-Refactor サイクルをAIに強制:**
  AIは一気に実装から書きがちなので、ルール（`RULES.md`）で「必ず先に失敗するテストを書くこと」を強制する。
- **境界値・異常系のテストケース列挙:**
  AIは正常系の実装が得意だがエッジケースを見落とすことがある。そのため、「仕様を元に、正常系・異常系・境界値のテストケースをリストアップして」とAIに依頼すると網羅性が上がる。
- **モックとスタブの活用:**
  外部依存がある部分はAIにモック化のコードを書かせることで、テストを決定的にし、実行を早くする。
- **仕様とテストのトレーサビリティ:**
  仕様書（`AC-*`）とテストケース（`TC-*`）をリンクさせる（`TEST_PLAN.md`など）。AIが「このコード変更はどの仕様を満たすためか」を把握しやすくする。

## 💡 4. 既存プロジェクトからのテンプレート流用検討

現在の `aidle` 開発プロジェクト自体が、上記のベストプラクティスを体現した素晴らしい構成になっています。このプロジェクトで運用されているドキュメント群の構造を、そのまま `templates/default/` 配下のテンプレートとして流用・汎用化することが可能です。

### 流用可能な構成要素

#### `AGENTS.md` (正本)
- **流用ポイント:**
  - AIに対する「動作の基本方針（Goal）」
  - 「TDD Cycle (必須)」の強制（Red -> Green -> Refactor）
  - 「Work Rules（作業ルール）」「Definition of Done（完了の定義）」の明記
  - 合意なしで進めない「合意ゲート」の仕組み

#### `docs/RULES.md` (開発ルール)
- **流用ポイント:**
  - TDD、品質ゲート（カバレッジ目標など）、レビュー基準の項目
  - コミットメッセージのフォーマット規定
  - AIに「計画（Plan）」を作らせる手順の定義

#### `docs/AGENT_CONTEXT.md` (現在の文脈)
- **流用ポイント:**
  - `現在フェーズ`、`優先ドキュメント読了順`、`次アクション` といったフォーマット
  - AIのコンテキストを最新状態に保つための「状態管理」としての役割

#### `docs/SPEC.md` (仕様書)
- **流用ポイント:**
  - `目的` `対象ユーザー` `スコープ（In/Out）` `ユースケース` `機能要件` の項目構成
  - 受け入れ基準を `AC-*` というIDで箇条書きにするフォーマット

#### `docs/TEST_PLAN.md` (テスト計画)
- **流用ポイント:**
  - `AC-* -> TC-*` のトレーサビリティ表
  - テストケース一覧を `TC ID`, `対応AC`, `観点`, `初期状態 (Red/Green)` で管理するマトリクス構造
  - 「実装順序」をAIに考えさせるセクション

#### `docs/TODO.md` (タスク・進捗管理)
- **流用ポイント:**
  - タスクを `Milestones` としてフェーズごとに管理する構造
  - 各マイルストーンに `達成条件` と `TODO` リストを設けるフォーマット
  - 「合意が取れていない項目は完了扱いにしない」というルール

#### `docs/KNOWLEDGE.md` (知見・ADR)
- **流用ポイント:**
  - アーキテクチャ決定記録（ADR）のフォーマット（Context, Decision, Consequences, Follow-ups）

## 🎯 次のステップ：テンプレート化の方針
これらの既存ドキュメントから「`aidle` プロジェクト固有の記述（RustやCLIに特化した部分）」を削ぎ落とし、どんなプロジェクトでも使える「汎用的なプレースホルダー」に置き換えることで、高品質な `default` テンプレートが完成します。
