# TEST_PLAN.md

本ドキュメントは、製品仕様書 (`SPEC.md`) の受け入れ基準 (`AC-*`) を検証するためのテスト計画を定義します。
AI駆動開発（AIDD）において、本ファイルは **「仕様と実装を繋ぐトレーサビリティの要」** となります。

## 🧪 テスト設計の原則

1. **AC-TC トレーサビリティ & 密度**: 1つの `AC-*` に対して、正常系 (Normal) に加え、境界値 (Edge) や異常系 (Error) を含む **多角的なテスト密度** を確保せよ。
2. **テストピラミッドの遵守**: テストの件数は **UT > IT > E2E > Live** の比率を維持せよ。
   - **UT**: ロジックの全分岐を網羅する（網羅性重視）。
   - **IT**: コンポーネント間の境界と状態遷移を検証する（連携重視）。
   - **E2E**: 主要なユーザーシナリオの導通を確認する（体験重視）。
3. **失敗モード分析 (FMEA)**: 実装前に「どのように壊れうるか」をリストアップし、それらを網羅するテストを設計せよ。
4. **Harness Verification**: すべてのテストは `scripts/check_harness.sh` で一括実行可能であれ。
5. **SDD & TDD**: テストコードは、AIに対する最も明確な「プロンプト」である。

## 📊 テストレベルと密度目標

プロジェクトの重要度（Tier）に応じて、以下の密度目標を達成すること。

| レベル | 対象 | 密度の指標 | 目標 (Tier 1) | 目標 (Tier 2) |
| :--- | :--- | :--- | :--- | :--- |
| **UT (Unit)** | 内部ロジック | **分岐網羅 (Branch)** | 100% | 80% 以上 |
| **IT (Integration)** | 外部連携/状態 | **インターフェース/状態網羅** | 全パターン | 主要パターン |
| **E2E (Scenario)** | ユーザー体験 | **AC 充足率 / シナリオ** | 全AC網羅 | 主要AC網羅 |
| **Live (Verify)** | 本番環境導通 | **SLO / 外形監視** | 必須 | 任意 |

## 🛡️ 失敗モード分析 (FMEA)

aidle が直面しうる失敗モードと、それに対するテスト戦略を定義します。

| 失敗モード | 影響 | 対策 / 検証テスト |
| :--- | :--- | :--- |
| **無効なテンプレート名** | 初期化不能、エラー | `tc017`, `tc029`, UT: `resolve_template_source` |
| **出力先ディレクトリ権限なし** | 作成不能、エラー | `tc007`, UT: `create_required_files` (error case) |
| **既存ファイルとの競合** | 上書きによる損失 | `tc010`, `tc011`, `tc013`, UT: `force_overwrite` |
| **不正な Markdown 構造** | 比較 (`check`) の誤判定 | `tc031`, UT: `extract_headings` |
| **不完全なテンプレートセット** | 生成物の欠落 | `tc019`, UT: `load_template_files` |
| **部分的な生成失敗** | 中途半端な状態 | `tc008`, `tc009`, UT: `rollback_state` |
