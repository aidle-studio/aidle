# テンプレート・アーキテクチャ設計提案（MS14）

## 1. 背景と目的
OpenAIの「Harness Engineering」の事例および「3本柱（Harness, SDD, Context）」の概念に基づき、AIエージェントが最大のパフォーマンスを発揮するための「エージェントファーストなプロジェクト環境」を設計します。
本設計は `aidle` のテンプレートとして出力されるディレクトリ構造とルールの仕組みを再定義するものです。

## 2. コアコンセプト
1. **段階的開示 (Progressive Disclosure)**
   - `AGENTS.md` は巨大なプロンプトではなく、100行未満の「地図（ポインタ）」に留めます。
2. **システム・オブ・レコード (System of Record)**
   - 人間の脳内やチャットではなく、リポジトリ内の `docs/` を唯一の情報源とします。
3. **機械的検証と自己修復レール (Harness)**
   - ルール違反を自然言語で諭すのではなく、検証スクリプトやリンターで弾き、エラーメッセージに「エージェントへの修正指示」を埋め込むことで自己修復ループを作ります。

## 3. 提案するディレクトリ構造 (`default` テンプレート)

```text
/ (Project Root)
├── AGENTS.md                  # 最上位の目次。各ドキュメントへのポインタと基本プロンプト（100行未満）
├── docs/                      # システム・オブ・レコード（正本）
│   ├── AGENT_CONTEXT.md       # 状態管理（現在のフェーズ、直近ゴール、次アクション）
│   ├── ARCHITECTURE.md        # プロジェクトのシステム設計・依存関係・階層構造の定義
│   ├── SPEC.md                # SDDの要。要件 (REQ-*) と受け入れ基準 (AC-*)
│   ├── TEST_PLAN.md           # AC-* と TC-* のトレーサビリティ・マトリクス
│   ├── RULES.md               # 開発ルール。長文を廃止し「検証スクリプトを通すこと」をDoDとする
│   ├── KNOWLEDGE.md           # ADR、過去の失敗と対策（例外運用記録など）
│   └── plans/                 # MS（マイルストーン）ごとの実行プラン (例: YYYY-MM-DD_MS*.md)
├── scripts/                   # Harness（機械的検証レール）
│   └── check_harness.sh       # エージェントが実行すべき完了条件チェッカー（Lint, Test, 形式チェック）
└── .github/hooks/             # (任意) commit-msg フックなどによる早期エラーフィードバック
```

## 4. ルール（Harness）の機械的検証手法
これまでの「AIにお願いするルール」から「ツールが強制する制約」へと移行します。

- **`check_harness.sh` の導入:**
  エージェントがタスクを「完了（Done）」とする前に必ず実行しなければならないスクリプトを同梱します。
  このスクリプトは、リンター、フォーマッター、テストランナーを統合して実行します。
- **AI向けのエラーメッセージ:**
  テストやLintが失敗した場合、コンソール出力として「AIが次にどう行動すべきか（例: "ルール違反です。`docs/RULES.md` の第3項を確認し、`src/` 内の依存関係を修正してください"）」を出すように検証ツール（またはラッパー）を構成します。
- **SDDとトレーサビリティの強制:**
  可能であれば、`TEST_PLAN.md` の `AC-*` に対応する `TC-*` が存在するかを正規表現でチェックする仕組みを組み込みます。
