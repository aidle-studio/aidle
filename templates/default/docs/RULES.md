# RULES.md

このファイルは、プロジェクト固有の開発ルールや品質基準を定義する「憲法」です。
AIエージェントおよび開発者は、必ずこのルールに従って開発を進めてください。

## 1. 開発原則
- **仕様駆動開発（SDD）**: 仕様は `docs/SPEC.md` を正本とし、`REQ-*`（要求）と `AC-*`（受け入れ基準）で管理する。
- **テスト駆動開発（TDD）**: `Red -> Green -> Refactor` を必須サイクルとする。
- **合意ゲート（Agreement Gate）**: AIが自律的に仕様変更や破壊的変更を行うことは禁止する。必ず提案を作成し、人間の開発者と合意した後に実装へ進める。

## 2. AI駆動開発の標準手順
1. マイルストーンを定義し、達成タスクへ細分化する。
2. マイルストーン達成のためのプランを作成し、合意を得る。
3. 合意したプランに従って実行する（都度進捗を報告する）。

## 3. コンテキスト更新ルール
- セッションの開始時または状態に変化があった場合、AIエージェントは自律的に `docs/AGENT_CONTEXT.md` の「現在フェーズ」と「直近ゴール」を最新状態に更新すること。
- 次回のアクションが明確になった時点で、「次アクション」のリストを更新し、コンテキストの連続性を維持すること。

## 4. プラン保存ルール
- 複雑な機能実装やリファクタリングを行う際は、事前にプランを作成し `docs/plans/` 配下にファイルとして保存する。
- ファイル名は `YYYY-MM-DD_<milestone-id>.md` 形式を推奨する。
- 途中再開時は、同一プランファイルを更新して履歴を残す。

## 4. TDDサイクル（必須）
1. **Red**: 失敗するテストを先に追加する（AIはこれを省略してはいけない）。
2. **Green**: テストを通す最小実装を追加する。
3. **Refactor**: 挙動を変えずに整理する。

## 5. テストルール
- テストケースは `docs/TEST_PLAN.md` において `TC-*` として管理し、必ず仕様の `AC-*` と対応付ける。

- モックの活用: 外部APIやデータベース依存は極力モック化し、テストを高速かつ決定的にする。

## 6. 品質ゲート

- カバレッジ基準: `[TODO: 目標カバレッジ（例: 全体80%以上）があれば記載]`

## 7. レビュー基準
- 仕様（AC-*）・テスト（TC-*）・実装のトレーサビリティがあること。
- 破壊的変更は明示され、影響範囲が説明されていること。
- 複雑なロジックには技術的意図を示すコメントが存在すること。

## 8. コミット規約
- 基本形式は「タイトル1行」とする。
- タイトルは何を変更したかが分かる短い文にする（目安: 50文字前後）。
- バグ再現手順、設計トレードオフ、判断理由など特筆事項がある場合は、タイトル + 空行 + 本文の形式にする。
- 本文では「何を変えたか / なぜ必要か / 実装上の注意点」を明記する。

## 9. Definition of Done (完了の定義)
- 対応するテスト（`TC-*`）が Green である。
- 影響範囲のすべての既存テストが成功している。
- Linter / Formatter がエラーなく通過している。
- 仕様（`SPEC.md`）との差分がない（必要ならドキュメント更新済み）。
- 変更理由がコミットまたはPR説明で追跡可能である。
