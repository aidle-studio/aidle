# SECURITY.md

本ドキュメントは、プロジェクトのセキュリティ基準、機密情報の取り扱い、および安全なコーディングの指針を定義します。

## 🛡️ セキュアコーディング原則 (Secure by Design)

本プロジェクトでは、環境（Web、CLI、インフラ等）を問わず、以下の普遍的なセキュリティ原則に従って設計・実装を行ってください。

1. **最小権限の原則 (Least Privilege)**: プログラムやコンポーネントには、そのタスクを遂行するために必要な「最低限の権限」のみを付与せよ。
2. **ゼロトラスト (Zero Trust)**: ユーザー入力、外部APIのレスポンス、環境変数、ファイルなど、外部からのデータはすべて「悪意がある（Untrusted）」と仮定し、必ず検証・無害化（サニタイズ）せよ。
3. **安全な失敗 (Fail Securely)**: エラーや例外が発生した際は、システムを安全な状態にしてから停止せよ。機密情報（APIキー等）や内部構造を含むスタックトレースをログ、画面、コミットに決して露出してはならない。

## 🤖 AIエージェント向け要注意パターン (Requires Agreement)

AIエージェントは、以下のパターンを**「強く非推奨 (Avoid)」** として扱ってください。
どうしても使用する必要がある場合は、実装前に人間にリスクと代替案を説明し、**明示的な合意（Agreement Gate）** を得なければなりません。

- **OSコマンド/シェルインジェクション**: `eval`, `exec` 等を用いた動的なコード実行や、ユーザー入力を直接結合したシェルコマンドの実行。（*ゼロトラスト違反*）
- **機密情報の平文出力**: デバッグ目的であっても、ログやエラーメッセージにパスワードやAPIキーなどのシークレットをそのまま出力すること。（*安全な失敗違反*）
- **パストラバーサル**: ユーザー入力を直接用いたファイルパスの構築や、システム領域への無制限なアクセス。

## 🔒 セキュリティ・スキャン

エージェントは、以下のセキュリティチェックをハーネス（検証レール）の一部として構築してください。

- **静的解析 (SAST)**: コード内の脆弱性を検知。
- **依存関係スキャン**: 既知の脆弱性があるパッケージの検知。
- **シークレットスキャン**: 誤ってコミットされた秘密情報の検知。

## 🚨 脆弱性発見時の対応

セキュリティ上の懸念を発見した場合は、速やかに `SECURITY.md` に追記し、人間へ報告してください。
公開されたIssueとして記録する前に、非公開の連絡ルートの有無を確認してください。
