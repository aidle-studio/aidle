# AGENTS.md

このプロジェクトは **テスト駆動開発（TDD）** を標準プロセスとして進めます。  
すべての実装・修正は、以下のルールに従ってください。

## 🚀 開発の進め方

本プロジェクトは **厳格なテスト駆動開発 (TDD)** に基づいて進行します。

```text
仕様検討 (SPEC.md) ➔ テスト計画 (TEST_PLAN.md) ➔ テスト実装 ➔ 機能実装 ➔ 検証 (Lint / Test)
```

## 🗺️ プロジェクトの地図

開発に関する重要な情報はすべて `docs/` ディレクトリに集約されています。  
新しいエージェントや開発者は、まず以下のドキュメントから順に読み進めてください。

1. **[AGENT_CONTEXT.md](docs/AGENT_CONTEXT.md)**: AIエージェント向けの最重要コンテキスト。プロジェクトの現状と行動指針が記されています。
2. **[RULES.md](docs/RULES.md)**: 開発の「憲法」です。コーディングスタイル、TDDサイクル、品質基準が記載されています。
3. **[SPEC.md](docs/SPEC.md)**: アプリケーションの機能仕様書。
4. **[TODO.md](docs/TODO.md)**: 現在の進捗とロードマップ。
5. **[TEST_PLAN.md](docs/TEST_PLAN.md)**: テストの計画と実施記録。
6. **[KNOWLEDGE.md](docs/KNOWLEDGE.md)**: 過去の決定事項や技術的な知見（ADR）。

## 🎯 Goal
- 仕様をテストで明確化し、最小実装で通し、継続的にリファクタリングする。
- 振る舞いの回帰を防ぎつつ、変更しやすい設計を維持する。

## 🔄 TDD Cycle (必須)
1. **Red**: まず失敗するテストを書く。
2. **Green**: テストを通すための最小限の実装を行う。
3. **Refactor**: テストが通っている状態を維持したまま整理する。
4. 1〜3を小さい単位で繰り返す。

## 🛡️ Work Rules
- 新機能・バグ修正は、原則として先にテストを追加する。
- 1コミットあたりの変更は小さく保つ（テストと実装の対応を明確にする）。
- 不要な機能の先回り実装はしない（YAGNI）。
- 既存テストを壊す変更は、意図と影響範囲を明示する。
- **仕様検討で決めた内容は、実装前に必ず人間の開発者と合意する（Agreement Gate）。**

## 🧪 Test Design Guidelines
- テスト名は期待される振る舞いを記述する。
- 1テスト1関心に寄せ、失敗理由が一目で分かるようにする。
- 外部依存はモック/スタブで分離し、テストを決定的にする。
- 境界値・異常系を優先的にカバーする。

## ✅ Definition of Done
- 追加/変更した仕様を表すテストが存在する。
- すべてのテストがローカルで成功する。
- 必要なリファクタリングが完了している。
- 変更理由がコミットメッセージまたはPR説明で追跡できる。

## 🤝 Commit / PR Policy
- 推奨順序:
  1. 失敗するテスト追加
  2. 最小実装
  3. リファクタリング
- PRには以下を含める:
  - 何を保証するテストか
  - 想定した失敗ケース
  - 残課題（あれば）

## 🤖 Agent Instructions (AIへの指示)
- 変更作業時は、まずテスト追加/更新案を提示し、次に実装する。
- テスト実行コマンドと結果を簡潔に報告する。
- テストがない変更は例外扱いとし、理由を明記する。
- 仕様検討フェーズでは、提案内容ごとに合意確認を取り、**合意済みの事項のみを次工程へ進める**。
