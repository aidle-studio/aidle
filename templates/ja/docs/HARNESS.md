# HARNESS.md

本ドキュメントは、AIエージェントがプロジェクト環境を **自律的に設営し、品質を自己修復レールに乗せるためのガイドライン** です。
セッション開始時に必ず内容を確認し、自分のための検証ツールやフックが正しく機能しているかを保証してください。

## 🤖 エージェントの任務：ハーネス設営

プロジェクトに参画した直後、または環境の変化を検知した際は、以下の「ハーネス（補助輪）」を自律的にセットアップしてください。

### 1. 高速フィードバックの構築
- **高速リンターの選定**: プロジェクトの言語（Rust, TS, Python...）に合わせ、最も高速なツール（Oxlint, Ruff, Biome等）をインストールし、設定せよ。
- **自動修正フックの構成**: エージェントが保存（Save）した直後に自動でフォーマットとLint修正が走るよう、エディタ設定や Git Hooks（`pre-commit`等）を構築せよ。

### 2. ルール・設定ファイルの保護（Test Cheatingの防止）
AI自身が「設定を緩める」ことでエラーを強引に回避する（ズルをする）ことを防ぐため、以下のガードレールを設置せよ。
- **監視対象**: `.eslintrc`, `rustfmt.toml`, `nextest.toml` などの設定ファイル、および既存のテストコード。
- **ガードレール例**: これらのファイルが変更された場合、コミット前に必ず Warning を出し、**「人間（ユーザー）の明示的なレビューと合意」**を要求する仕組みを構築すること。

### 3. セキュリティスキャン (Secret & SAST)
誤って機密情報（APIキー、パスワード）をコミットしてしまうのを防ぐため、**ローカルで回せる軽量なシークレットスキャン（`gitleaks`、`trufflehog`、またはLinterのカスタムルールなど）を `check_harness.sh` や Pre-commit hook に組み込むこと**を強く推奨します。

### 4. 完了条件チェッカー (`check_harness.sh`) の実装
プロジェクトのルートに `scripts/check_harness.sh` を作成し、以下の「品質ゲート」を一括実行するロジックを実装せよ。
- **Lint & Format**: 警告ゼロ、差分ゼロであること。
- **Test**: 全てのテストが Green であること。
- **Coverage**: `TEST_PLAN.md` で定義されたカバレッジ目標（例: Branch 100%）を満たしているか、CIツール（`llvm-cov`, `jest --coverage`等）を用いて厳格にチェックすること。
- **出力**: 全て成功した時のみ `SUCCESS`（または `Done`）と出力し、1つでも失敗すれば即座に `exit 1` で終了させること。

## 🛠️ 検証コマンドの期待値

エージェントは、実装中に以下のコマンド（またはそれと同等の統合コマンド）が常に `SUCCESS` を返す状態を維持する義務があります。

1. **Lint**: `cargo clippy`, `eslint`, `ruff check` 等（警告ゼロ推奨）
2. **Format**: `cargo fmt`, `prettier`, `black` 等（修正差分ゼロ）
3. **Test & Coverage**: `cargo llvm-cov`, `jest --coverage` 等（閾値クリア）
4. **Harness**: `scripts/check_harness.sh`（一括チェック）

## 🔄 自己修復のフロー

検証に失敗した場合、AIは以下のステップで自律的に解決を図ってください。

1. エラーメッセージから「違反箇所」と「原因」を特定する。
2. ハーネスが提供するヒントや、関連ドキュメント（RULES.md, RELIABILITY.md）を参照して修正方針を立てる。
3. 修正を適用し、再度検証スクリプトを実行して `SUCCESS` を確認する。
