# TEST_PLAN.md

本ドキュメントは、製品仕様書 (`PRD.md`)、基本仕様書 (`ARCHITECTURE.md`)、および機能仕様書 (`docs/features/*.md`) で定義された「要件・制約・受け入れ基準」を検証するためのテスト計画の指針であり、エージェントがテストを実装する際に必ず遵守すべき**「テストの法律」**です。

## 🧪 テスト設計の原則と禁止事項

テストの件数は **UT > IT > E2E** のピラミッド比率を維持してください。
エージェントはテストを書く・実行する際、以下の原則と**禁止事項**に必ず従うこと。

1. **水増しとサボりの排除**:
   - **❌ 厳禁 (Assert-free Test)**: 分岐網羅率を稼ぐためだけに、結果の検証（`assert`等）を行わず単に関数を呼び出すだけの無意味なテストを書いてはならない。
   - 意味のない重複テストや、単に「件数を稼ぐため」のテストは固く禁ずる。
2. **テスト改ざんの禁止 (No Test Cheating)**:
   - **❌ 厳禁**: プロダクトコードの実装中にテストが失敗した際、**「テストコードの期待値を書き換えて無理やり通す」ことは絶対に行ってはならない**。テストを修正して良いのは、機能仕様（AC）そのものが変更された時のみである。
3. **多角的なテスト (FMEAと境界値の徹底)**:
   - 正常系だけでなく、境界値（空、Null、最大値、絵文字/特殊文字、極端に長い文字列等）を必ず含めること。
   - 結合テスト（IT）では、**「タイムアウト」「接続エラー」「不正データ」などの故障モード観点**を必須とする。
4. **決定論的なテスト**:
   - ネットワーク、現在時刻、乱数などの外部要因に依存せず、常に同じ結果を返すようにモックやスタブを活用せよ。

## 🧪 ソフトウェア品質特性に基づく必須テスト観点

単に「動くか」だけでなく、ISO/IEC 25010 に基づく以下の品質特性を網羅してテストを設計してください。

### 1. 機能適合性 (Functional Suitability)
「仕様通りに正しく動くか」を検証する最も基本的なテスト。
- **機能完全性**: 仕様書 (`PRD.md`, `features/*.md`) に記載された全てのユースケースが実行可能か。必要なデータが正しく保存・更新・表示されるか。
- **機能正確性**: 計算結果が期待値と完全に一致するか。境界値（最小値、最大値、その前後、空、Null）で期待通りの判断がなされるか。
- **機能適切性**: ユーザーが目的を達成するまでのステップに、無駄な操作や矛盾はないか。

### 2. パフォーマンス効率性 (Performance Efficiency)
「リソースをどう使っているか」を検証するテスト。
- **時間効率性**: 応答速度は目標値（例: 1秒以内）に収まっているか。大量データ処理時（N+1問題の有無など）に時間が指数関数的に増大しないか。
- **資源効率性**: メモリ使用量やCPU負荷が、処理終了後に元のレベルに戻るか（メモリリークの防止）。
- **容量満足性**: 負荷がかかった際、エラーにならずにキューイング等で耐えられるか。

### 3. 信頼性 (Reliability)
「過酷な状況でどう耐えるか」を検証するテスト（※IT層で必須）。
- **成熟性**: 通常の操作範囲で、理由の不明なクラッシュやハングアップが起きないか。
- **障害許容性 (Fault Tolerance)**: 外部APIやDBが停止している際、システム全体が落ちずに「フォールバック」や「エラー表示」で耐えられるか。
- **回復性**: 異常終了後、再起動した際にデータが破損せず、直前の状態から復旧できるか（トランザクションのロールバック）。

### 4. 使用性 (Usability)
「ユーザーが迷わないか、自滅しないか」を検証するテスト。
- **運用性**: エラーメッセージの内容は具体的で、ユーザーが「次に何をすべきか」理解できるか。
- **ユーザーエラー保護**: 誤った入力（文字種違い、極端に長い文字列等）を即座に弾き、安全な状態を保てるか。

### 5. セキュリティ (Security)
「悪意ある操作への耐性」を検証するテスト。
- **機密性**: 権限のないユーザーが他人のデータにアクセスできないか（認可の検証）。
- **完全性**: 外部からの入力値（SQLインジェクション、XSS等）を無害化して処理しているか。
- **否認防止**: 「誰がいつ何をしたか」というログ（Trace ID等）が確実に残っているか。

## 📊 重要度 (Tier) の定義と密度目標

プロジェクトの信頼性を担保するため、タスクの重要度（Tier）に応じて以下の**最低限のテスト密度**を達成しなければなりません。プラン作成時（`PLAN_TEMPLATE.md`）に、この基準に基づきテストケースを洗い出してください。

| Tier (重要度) | 定義 | 密度目標: 1要件(AC)あたりのケース数 | 網羅率目標 (Branch) |
| :--- | :--- | :--- | :--- |
| **Tier 1 (Core)** | システムの根幹ロジック、データ整合性、セキュリティ等 | **平均 5.0 ケース以上**<br>(例: 正常1, 境界2, 異常2) | **100%** |
| **Tier 2 (Feature)** | 主要なユーザー機能、ビジネスロジック等 | **平均 3.0 ケース以上**<br>(例: 正常1, 境界1, 異常1) | **80% 以上** |
| **Tier 3 (Utility)** | 補助的なツール、内部ユーティリティ、UI微調整等 | **平均 1.0 ケース以上**<br>(正常系のみでも可) | 主要パス導通 |

## 🛡️ 各テストレベルの役割と検証観点

エージェントは、以下の役割分担に従ってテストを配置してください。

### UT (Unit Test)
- **役割**: 最小単位（関数・モジュール）のロジックを高速に検証する。
- **必須観点**: 同値分割、境界値分析（最大・最小・空・ゼロ・不正な型など）。
- **目標**: すべての分岐（if/else, match等）をここで叩き潰す。

### IT (Integration Test)
- **役割**: コンポーネント間の境界（DB連携、外部API呼び出し等）を検証する。
- **必須観点**: 相手が壊れた場合（Timeout, 5xx Error, 切断, スキーマ不一致）の回復・フォールバック動作。

### E2E (End-to-End Test)
- **役割**: 主要なユーザーシナリオが最初から最後まで通るか（導通）を検証する。
- **必須観点**: ハッピーパス、および代表的なユーザーの暴走（連打、中断等）。
