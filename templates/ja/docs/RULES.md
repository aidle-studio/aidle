# RULES.md

本プロジェクトの開発プロセスと完了条件（DoD）を定義する「憲法」です。
AIエージェントおよび開発者は、必ずこのルールに従って開発を進めてください。

## 1. 開発原則（Harness-First）

- **検証駆動**: 自然言語の指示よりも、`scripts/check_harness.sh` などの機械的検証を優先し、信じてください。
- **仕様駆動 (SDD)**: `docs/product-specs/` の要件 (`REQ-*`) と受け入れ基準 (`AC-*`) を唯一の正本とします。
- **テスト駆動 (TDD)**: `Red -> Green -> Refactor` のサイクルを最小単位として繰り返してください。

## 2. 標準開発プロセス

1. **計画 (Plan)**: `docs/exec-plans/active/` にプランを作成し、人間と合意（Agreement Gate）する。
2. **ハーネス確認 (Harness)**: `docs/HARNESS.md` に従い、必要な検証レールが敷かれているか確認する。
3. **実装 (Act)**: 仕様とテストに基づき、最小限のコードを実装する。
4. **レビュー (Review)**: 以下の「自己レビューチェックリスト」に従い、自身の成果物を客観的に評価し、必要に応じて修正する。
5. **検証 (Verify)**: `scripts/check_harness.sh` を実行し、機械的な合格を確認する。

## 3. 自己レビューチェックリスト

エージェントは、検証 (Verify) の前に以下の 4 つの視点で自己レビューを実施してください。

### 📦 ソースコード (Source Code)
- [ ] **命名**: 変数・関数名は意図を明確に表しているか？
- [ ] **安全性**: エッジケース、ヌルポインタ、例外処理は考慮されているか？
- [ ] **凝集度**: 1つの関数やモジュールが大きすぎないか？責務は適切か？

### 🧪 テストコード (Test Code)
- [ ] **密度**: `TEST_PLAN.md` の Tier 別目標（分岐網羅率等）を満たしているか？
- [ ] **多角性**: FMEA（失敗モード分析）に基づき、境界値や異常系が網羅されているか？
- [ ] **独立性**: テスト間で状態を共有せず、単独で実行可能か？

### 🤖 ハーネス (Harness)
- [ ] **自動化**: `check_harness.sh` で今回の追加分も正しく検証されるか？
- [ ] **フィードバック**: 検証失敗時に、修正のヒントとなる明確なログが出るか？

### 📝 ドキュメント (Docs)
- [ ] **整合性**: `SPEC.md`, `ARCHITECTURE.md` 等、関連ドキュメントとの乖離はないか？
- [ ] **明確性**: 実装の「なぜ」が、後で読む人間や他のエージェントに伝わるか？

### ✨ プロダクトセンス (Product Sense) - Tier 2 以上推奨
- [ ] **ユーザー体験**: エラーメッセージは親切か？（「エラー」ではなく「解決策」を提示しているか）。
- [ ] **意図の合致**: `PRODUCT_SENSE.md` で定義されたターゲット層の期待に沿う挙動になっているか。

### 💎 アーキテクチャ・設計 (Design) - Tier 1/2 必須
- [ ] **整合性**: `ARCHITECTURE.md` の設計方針と矛盾していないか？
- [ ] **結合度**: 既存コンポーネントと密結合になりすぎていないか？再利用性は考慮されているか？

### ⚡ 性能・最適化 (Performance) - Tier 1 必須
- [ ] **効率**: 明らかなボトルネック（計算量の増大、不必要な N+1 クエリ等）はないか？
- [ ] **リソース**: メモリやネットワーク資源を無駄に消費していないか？

## 4. コンテキスト更新ルール (Context Sync & GC)

セッション開始時、またはタスクの状態に変化があった際は、以下のファイルを **アトミック（同時かつ整合性を保って）に更新** しなければなりません。

1. **`docs/AGENT_CONTEXT.md`**: 「現在フェーズ」「直近ゴール」「次アクション」を最新化する。
2. **`docs/TODO.md`**: 完了したタスクのチェックボックス `[x]` を埋め、進捗サマリーを更新する。
3. **`docs/exec-plans/active/*.md`**: 実行中のプランのステータスやログを更新する。
4. **`docs/KNOWLEDGE.md`**: バグ修正や重要な教訓を得た場合は、必ず追記する。

**🧹 完了プランの断捨離 (Garbage Collection)**
マイルストーンやタスクが完了（Done）した際は、コンテキストの肥大化を防ぐため、以下の手順を必ず実行してください。
- 実行完了したプランファイルを `docs/exec-plans/active/` から `docs/exec-plans/completed/` に移動（`mv`）する。
- 移動の際、そのプランで得た「次以降に活かせる重要な教訓やハマりポイント」を抽出し、`docs/KNOWLEDGE.md` に転記する。

*※これらの同期・断捨離が完了していない状態での「タスク完了」の宣言は認められません。*

## 5. 重要度 (Tier) の定義

全ての開発タスクは、その影響範囲に応じて以下の Tier に分類され、求められるテスト密度が異なります。

- **Tier 1 (Core)**: システムの根幹ロジック、データ整合性、セキュリティに関わる機能。
- **Tier 2 (Feature)**: 主要なユーザー機能、ビジネスロジック。
- **Tier 3 (Utility)**: 補助的なツール、内部ユーティリティ、UIの微調整。

## 6. 品質ゲート（Definition of Done: DoD）

タスクの「完了（Done）」を宣言するには、以下の条件をすべて満たす必要があります。

- **仕様充足**: 対応する `AC-*` にリンクしたテスト (`TC-*`) がすべて `Green` である。
- **自己レビュー完了**: 「自己レビューチェックリスト」の全項目を確認し、問題がないと判断している。
- **テスト密度の確保**: `docs/TEST_PLAN.md` で定義された Tier 別の密度目標を満たしている。
- **ハーネス成功**: `scripts/check_harness.sh`（Lint, Format, Test等の一括検証）が成功している。
- **合意完了**: 破壊的変更がある場合、その影響範囲が説明され、合意されている。
- **最新化**: `docs/TODO.md` および `docs/AGENT_CONTEXT.md` が最新の状態に更新されている。

## 7. プロセスの進化 (Process Evolution)

本プロジェクトのルール（`RULES.md`, `TEST_PLAN.md` 等）は不変ではありません。開発を通じて発見された改善点は、以下の手順で自律的に反映させてください。

1.  **気づき (Detection)**: 開発中にプロセスの冗長性、矛盾、または新たな品質向上のアイデアを発見した際、直ちに「プロセス改善」のタスクを提案せよ。
2.  **提案 (Proposal)**: `docs/adr/` または実行プラン内で、ルールの変更内容とその理由（なぜ品質や速度が向上するか）を記述せよ。
3.  **反映 (Application)**: 合意（Agreement Gate）を得た後、関連するドキュメントおよび `templates/` を更新せよ。

## 8. コミット規約

- **タイトル**: 1行、何を変更したか簡潔に。
- **本文**: 必要に応じて「なぜ変えたか」「実装上の注意点」を記述。
- **トレーサビリティ**: コミットメッセージに関連する `AC-*` やマイルストーンIDを含めることを推奨します。
