# RELIABILITY.md

本ドキュメントは、プロジェクトの信頼性、テスト方針、および異常系・境界値への対応指針を定義します。

## 🛡️ エラーハンドリング戦略 (Resilience)

システムは「壊れないこと」よりも「壊れても安全に回復・停止すること」を優先して設計せよ。

1. **Fail-fast (早期失敗)**:
   - 不正な入力や予期せぬ状態を検知した場合は、処理を継続せず、即座にエラーを返すかパニック（停止）させよ。隠れたバグの温床を作らないこと。
2. **Retry (再試行)**:
   - ネットワーク通信やDB接続など、一時的な障害（Transient Error）が想定される処理には、必ず Exponential Backoff などを伴うリトライロジックを実装せよ。
3. **Fallback (代替手段)**:
   - 外部サービスが完全にダウンした場合でも、ユーザー体験を損なわないよう、キャッシュの返却やデフォルト値の適用など、適切なフォールバック経路を設計せよ。

## 🛡️ テストピラミッド

エージェントは、以下の比率を意識してテストを構築してください。

1. **Unit Tests (大半)**: ロジックの最小単位を検証。
2. **Integration Tests (中間)**: コンポーネント間の連携を検証。
3. **E2E Tests (少数)**: 主要なユースケースの疎通を検証。

## 🧪 テスト設計の原則

- **AC-TC トレーサビリティ**: 全ての受け入れ基準 (`AC-*`) に対して、少なくとも1つのテストケース (`TC-*`) を用意せよ。
- **異常系への配慮**: 正常系だけでなく、エラーケース、タイムアウト、リトライ、境界値を網羅せよ。
- **決定論的なテスト**: 外部要因（ネットワーク、現在時刻等）に依存せず、常に同じ結果を返すようにモックやスタブを活用せよ。

## 🚦 タイムアウトとパフォーマンス

- テストランナーでの実行時間は、可能な限り短く保つこと。
- 重いテストや長時間かかる処理は、専用のタグを付け、並列実行可能なように構成せよ。
